# csharp-winforms-guarded-dev

## 目标

这个 skill 用于跨项目的 **C# WinForms 开发与改造**，重点守住“先看项目、再写代码、贴近既有风格”的执行方式。

适用场景包括：

- 新增或修改 C# WinForms 窗体、维护页、配置页、查询页、列表页、弹窗。
- 给窗体接初始化、回显、校验、保存、查询、过滤、统计、事件逻辑。
- 对接 `BLL`、`IDAL`、`DALFactory`、`SQLServerDAL`、`Model` 等常见三层结构。
- 参考旧 VB 或旧 C# 业务逻辑迁移，但新代码落回当前项目的 C# 结构。
- 按当前项目规范复用自定义控件、公共转换、公共校验、公共提示、公共导出等能力。

## 强制边界

在这个 skill 生效时，遵守以下规则：

1. 新开发只输出 C#；旧 VB 只能作为业务参考，不延续为新代码语言。
2. 不生成总结性 Markdown 文档，不生成测试脚本，不主动补测试代码，除非用户明确要求。
3. 不编译，不运行，不在收尾处提示“你可以执行以下命令编译/运行”，除非用户明确要求。
4. 先检索项目上下文，再改代码；禁止脱离现有结构拍脑袋生成整套实现。
5. 所有层级的字段访问、数据绑定和业务转换，默认不要额外添加字段存在性、行数、路径等兜底判断；如果当前流程按规范必须返回某个字段、行或数据，就直接按规范访问，让缺字段、错字段、错数据在原位置暴露，避免静默掩盖真实问题。只有用户明确要求兼容脏数据、历史缺字段或外部不可信输入时，才增加有业务语义的防御处理。
6. 优先复用项目已有自定义控件、领域专用控件和公共方法，不重复造轮子。
7. UI 层只组织界面状态、输入值、筛选条件和流程调度；完整 SQL 主体默认下沉到数据访问层。
8. 数据访问默认走项目既有链路，如 `BLL` → `DALFactory` / `IDAL` → `SQLServerDAL`；不要另起并行访问链路。
9. 读取单字段、状态、存在性、聚合值时优先返回标量类型；只有多行多列结果集才返回 `DataSet` / `DataTable`。
10. 静态控件初始化和静态事件接线优先放在 `InitializeComponent()`；不要无依据新增 `BindEvents()` / `InitializeControls()`。
11. 构造函数默认只保留 `InitializeComponent()`；常规初始化、数据装载、默认值与界面行为放到 `Load` 或职责函数中。
12. 默认使用 4 空格缩进、Allman 花括号、明确命名、必要注释与明确异常信息；禁止吞异常。
13. 如果用户的需求混入编译、运行、自动测试、生成说明文档等未明确要求的内容，应主动收敛范围，只处理代码本身。

## 工作方式

### 1. 先理解项目上下文

开始修改前，优先检索并理解：

- 当前要修改的窗体、Designer、控件、BLL、IDAL/DAL、Model、SQL 脚本或配置文件。
- 是否已有相邻 C# 模块可复用。
- 是否存在旧 VB 或旧实现可作为业务参考。
- 当前项目是否已有规则文件、README、`.kilocode`、`.cursor` 或仓库内说明。
- 当前项目有哪些自定义控件、领域专用控件和公共方法可直接复用。

如果存在旧 VB 实现，只提炼业务流程、字段含义、事件顺序和校验逻辑；最终实现必须回到当前项目的 C# 结构中。

### 2. 再判断修改类型

输出代码前，先判断本次属于哪类修改：

- 窗体层修改：控件布局、事件、回显、校验、保存、界面行为。
- 业务层接入：调用 BLL、补保存/更新逻辑、查询数据。
- 分层联动修改：窗体 + BLL + IDAL/DAL + Model 一起改。
- 旧逻辑迁移：把历史 VB 或旧 C# 业务流程迁移成当前项目的 C# 写法。
- 查询/列表修改：筛选条件、Grid 列、统计、导出、主从联动。
- 字典/维护窗体修改：列表页、编辑页、扩展页、保存回传。

优先最小化改动范围；不要因为补一个字段或条件而顺手重写整页。

## 通用 WinForms / 三层开发基线

### 1. 命名规范

默认遵循：

- 类名、方法名、属性、公共字段、常量、枚举、事件使用 `PascalCase`。
- 局部变量、方法参数使用 `camelCase`。
- 私有字段使用 `_camelCase`。
- 常量使用 `UPPER_CASE`。
- Model 层常见命名为 `Mdl` + 实体名；BLL 为 `Bll` + 实体名；IDAL 为 `IDal` + 实体名；SQLServerDAL 为 `Dal` + 实体名。若项目已有不同规则，优先沿用项目规则。
- 字典维护窗体优先沿用 `Dict + 功能名 + 1/2/3`、`*1/*2/*3` 或当前模块既有命名，不自创新后缀体系。
- 控件命名默认使用“控件类型前缀 + 字段名 / 功能名”，如 `Txt`、`Num`、`Date`、`Dtcbo`、`Single`、`Btn`、`myGrid`、`Lbl`、`tableLayoutPanel`。
- 控件类型发生变化时必须同步改控件字段名，不保留旧类型前缀；例如文本框改固定项下拉后，`TxtPartCategory` 应改为 `SinglePartCategory`，不要继续使用 `Txt` 前缀。
- 固定项单选下拉（如 `MySingleComobo` 或其派生控件）优先使用 `Single` 前缀；数据源下拉按项目相邻习惯使用 `Combo` / `Dtcbo` 前缀；文本、日期、数值输入分别继续使用 `Txt`、`Date`、`Num` 前缀。
- 方法名优先使用“动词 + 名词”，常用前缀为 `Get`、`Set`、`Load`、`Save`、`Add`、`Update`、`Delete`、`Validate`、`Check`、`Init`。

### 2. 窗体基类、布局与 Designer

优先观察并贴近相邻模块：

- 窗体代码与 Designer 分工。
- 事件绑定位置。
- 初始化、回显、保存和查询入口。
- BLL 调用方式。
- DataRow / Model 回写方式。
- 自定义控件和领域控件的使用方式。

默认规则：

- 主窗体 / 列表窗体优先继承项目基类，如 `BaseFather`。
- 字典列表窗体优先继承项目字典列表基类，如 `BaseDict1`。
- 弹出 / 编辑窗体优先继承项目子窗体基类，如 `BaseChild`。
- 字典编辑窗体优先继承项目字典编辑基类，如 `BaseDict2`。
- 禁止无依据直接继承 `Form`；只有当前项目某类特殊窗体已有稳定例外且必须沿用时才允许。
- 布局默认优先使用 `TableLayoutPanel` 或相邻模块已有布局方式。
- 常见视觉基线可按行高 29px、单行输入控件高度 23、字体“宋体 10.5F”控制；若项目已有不同基线，优先沿用项目。
- 同一窗体或同一输入区域内，自定义输入控件的 `CaptainWidth` 应尽量保持一致；只有顶部过滤框、窄工具栏、跨列大备注等相邻模块已有特例时，才沿用特例宽度。
- Designer 中 `Captain` 标题不足四个汉字时，按相邻页面习惯用空格补齐到四字视觉宽度，例如 `编    码`、`名    称`、`价    格`、`备    注`；四个汉字标题保持原文，纯英文业务名保持原文，英文+中文混合标题可用空格分隔如 `HIS 编码`。
- 新增或拆分窗体时保持 `*.cs`、`*.Designer.cs`、`*.resx` 三件套配对存在。
- Designer 中只使用控件真实存在的属性，不擅自添加封装外属性。
- 对只展示不可编辑信息的自定义输入控件，是否在 Designer 中写死 `ReadOnly` / `Enabled` 应先看相邻模块；若项目倾向运行期控制，则在初始化或回显函数中统一设置，不在 Designer 中硬编码状态。
- 对系统自动生成、仅展示且用户不应编辑或获得焦点的字段，优先使用 `Enabled = false`，不要用 `ReadOnly = true`；典型场景包括拼音码、简码、简称等由名称自动生成的 `TxtJc` / `PinYin` 字段。只有业务明确需要控件仍可获得焦点、复制内容或保持启用态视觉时，才使用 `ReadOnly = true`。
- 参考现有窗体开发新窗体时，不要无依据增加参考窗体没有的结构、函数或控件属性。

### 3. 事件、入口与职责拆分

静态控件和窗体本身由 Designer 声明时，默认把静态事件接线写在 `*.Designer.cs` 的 `InitializeComponent()` 中：

- `Load`、按钮 `Click`、下拉 `RowChange` / `SelectedValueChanged`、Grid 选择变更等固定事件优先在 Designer 接线。
- 不要把本应由 Designer 管理的静态事件分散到窗体代码中重复绑定。
- 只有控件运行时动态创建、事件需要按条件订阅/取消订阅，或相邻模块已有动态绑定模式时，才手动绑定。

表单录入、维护、查询、配置、弹窗等界面优先保持职责分离：

- 初始化函数：负责控件装配、数据源绑定、事件挂接、界面行为设置。
- 清空函数：负责默认值、焦点、初始状态。
- 回显函数：负责把已有数据恢复到界面。
- 校验函数：负责判空与业务合法性检查。
- 模型装配函数：负责从控件取值并装配到 Model 或 DataRow。
- 保存入口：负责流程编排，不堆砌所有细节。
- 查询入口：负责收集条件、调用 BLL、绑定结果、刷新统计，不夹杂无关初始化。

### 4. 自定义控件与领域控件优先

只要项目内已有对应封装，优先使用项目控件而不是直接使用原生 WinForms 控件或第三方基础控件。

常见优先级：

- 文本输入优先项目 `MyTextBox` 等文本控件。
- 日期输入优先项目 `MyDateEdit` 等日期控件。
- 数值输入优先项目 `MyNumericEdit` 等数值控件。
- 固定选项下拉优先项目 `MySingleComobo` 等单选控件。
- 数据源下拉优先项目 `MyDtComobo` 等数据控件。
- 下拉容器优先项目 `MyDropDown` 等下拉容器控件。
- 多选输入优先项目 `MyMultiSelect` 等多选控件。
- 按钮优先项目 `MyButton` 等按钮控件。
- 列表/表格优先项目 `MyGrid`、`MyFlexGrid`、`MyGridChild` 等网格控件。
- 树优先项目 `MyTreeView` 等树封装或树节点传递模式。
- 时间范围优先项目 `DoubleDateEdit`、`DoubleDateEditH` 等双日期控件。
- 如果项目存在领域专用控件库，如部门、人员、角色、项目、仓库等业务选择器，优先使用领域专用控件，而不是手工拼通用下拉。

使用自定义控件时优先走其公开属性和事件，如 `Captain`、`CaptainWidth`、`Text`、`Value`、`SelectedValue`、`SelectedIndex`、`Enabled`、`ReadOnly`、`GotFocus`、`ValueChanged`、`SelectedValueChanged`、`RowChange`。不要绕过封装直接假设内部子控件结构，除非任务就是修改控件本身。

判空优先复用项目公共校验方法，如 `NotAllowEmpty(...)` 或相邻模块等价入口；不要把 `Text.Trim() == ""`、`SelectedValue == null` 等空值判断散落在按钮事件或多个函数中。业务合法性校验应写在判空之后，并集中在校验函数中处理。

### 5. 下拉控件规则

当界面包含数据源型下拉、固定项下拉或父子级联下拉时，先确认控件命名已从原输入控件前缀切换为下拉控件前缀；尤其是从文本框改成固定项下拉时，应把 `Txt*` 同步改为 `Single*`，让代码阅读者能从字段名直接判断控件类型。

优先按以下职责拆分：

- `FormInit()`：初始化下拉、设置固定项、配置显示参数、挂接事件。
- `DataClear()`：清空当前值、设置默认项、设置默认焦点。
- `DataShow(...)`：回显已有数据。
- `DataCheck()`：做必填校验与业务合法性校验。
- `FillModel(...)`：从控件取值并装配 Model 或 DataRow。
- `SelectedValueChanged` / `RowChange`：处理父子级联刷新。
- 保存按钮：只做流程调度，不堆砌控件细节。

编码、显示与保存规则：

- 数据源型下拉优先按编码字段通过 `SelectedValue` 回显与保存。
- 显示文本只用于界面展示或 DataRow 名称列同步，不要把显示文本直接写入编码字段。
- 编辑弹窗保存后回写父列表 `DataRow` 时，若列表数据源包含由查询联表或下拉展示产生的名称列（如 `*_Name`），在写入编码字段后同步用下拉 `Text.Trim()` 回写对应名称列；不要为了展示列把非持久化名称字段硬塞进 Model 或写入数据库。
- 固定项下拉优先按 `Text` 或 `SelectedIndex` 回显，避免编码值与显示值混用。
- 数据源型下拉清空优先使用 `SelectedIndex = -1`。
- 固定项下拉如需默认值，统一在清空函数中设置默认索引或默认文本。
- 判空优先复用项目公共校验方法，不把 `Text.Trim() == ""`、`SelectedValue == null` 之类判断散落在按钮事件中。
- 父子级联刷新后必须清空子控件旧值，避免保留脏选择。
- 输入法、焦点、默认选中等界面行为优先在初始化阶段统一挂接。

### 6. 查询 / 列表类窗体规则

查询页、列表页、统计页、主从表列表或维护页顶部过滤区，默认按“查询 / 列表类窗体”理解。

常见区域：

- 条件区：时间范围、科室、医生、状态、关键字、类型筛选。
- 按钮区：查询、清空/重置、导出、打印、明细。
- 结果区：主列表 Grid。
- 扩展区：从表、明细区、统计文本、页脚汇总。

默认规则：

- 新建查询窗体时，先稳定“条件区 + 主列表区”，再补导出、明细、从表联动。
- 查询初始化拆成主列表列初始化、查询控件默认值、下拉数据源、时间范围、列自定义/排序/分组/汇总、从表或明细区。
- Grid 优先调用项目已有初始化入口，再逐列配置；状态列如存在通常放第一列。
- 顶部过滤区若只需要关键字过滤，默认优先使用单个关键字框，不无依据增加“按字段切换”的下拉。
- 顶部区同时包含 toolbar、过滤框、统计标签、提示文本时，优先用 `TableLayoutPanel` 统一组织。
- 查询按钮只做流程调度：收集条件、组织筛选条件、调用 BLL、绑定结果、刷新统计。
- UI 层只允许组织 `WHERE` 条件片段，不直接拼完整 SQL 主体。
- 时间范围优先采用“开始时间 + 结束时间”，开始到 `00:00:00`，结束到 `23:59:59`，除非项目已有不同规则。
- 清空 / 重置按钮负责恢复条件默认值、清空关键字、重置下拉、恢复时间范围、清空主从表、恢复焦点。
- 主从表页面重新查询时应清空旧从表，避免沿用上一次查询残留明细。
- 导出、打印、明细按钮优先复用项目已有公共能力，不重新造同类工具。

### 7. 字典 / 维护窗体规则

当任务涉及基础字典、参数维护、系统配置、映射对照、主列表 + 编辑弹窗时，优先按字典 / 维护窗体模式理解。

常见分层：

- `*1`：列表页、树页、入口页，负责加载数据、选择当前行、过滤、打开编辑窗体。
- `*2`：新增/编辑页，负责 `FormInit`、`DataClear`、`DataShow`、`DataCheck`、`DataAdd`、`DataEdit`。
- `*3`：扩展页、子项页、附加配置页、明细页。
- `*Map1/*Map2` 或类似命名：对照关系维护页和编辑页。

默认规则：

- 新增一个字典维护时，优先思考是否应补齐 `1/2` 两个窗体，而不是只写一个 Form。
- 一级列表页通常需要初始化 Grid/Tree、绑定 DataTable/DataView/CurrencyManager、维护当前 DataRow、打开编辑页、处理回传。
- 二级编辑页通常在 `Load` 中初始化；新增时清空，编辑时回显；保存按钮先校验，再分流新增/编辑。
- 保存按钮只做流程调度，不直接写完整模型赋值、BLL 调用、列表回写、关闭窗体等全部细节。
- 取消按钮默认只关闭窗体，除非相邻模块已有确认、刷新或重置规则。
- 保存后同步更新当前列表行或回传对象，而不是要求用户手动刷新。
- 修改范围尽量落在当前字典目录；只加字段时同步检查列表显示、编辑回显、保存链路是否需要联动。

### 8. 公共能力与过度封装控制

若项目中已有公共能力，优先直接复用，不在窗体或局部流程中重复造一套等价封装：

- 公共提示、公共转换、公共空值处理、公共导出、公共拼音、输入法、焦点处理、权限判断、缓存读取等能力，优先走项目公共入口。
- 若项目存在 `Common/MessageHelper.cs`，WinForms 提示优先使用 `Common.MessageHelper`，如 `Common.MessageHelper.ShowWarning(this, "提示内容")`；显式传当前窗体作为 `owner`，不要在窗体内直接写 `MessageBox.Show(...)` 或再封装一次性私有 `ShowInfo()` / `ShowError()`。
- 从 `DataRow` / `DataTable` / `object` 等来源转换具体类型时，优先复用项目公共转换器，如 `Common.ConvertObject` 或相邻模块等价入口。
- 反向写入可空字段或处理 `NULL` / `DBNull` 时，优先复用项目公共空值处理入口，如 `Common.Tools.IsValueNull<T>(...)` 或相邻模块等价方法。
- 不新增 `ShowInfo()`、`ShowError()`、`GetInt()`、`GetString()`、`ExportGrid()` 这类只包一行公共调用的一次性私有函数，除非它额外承载明确业务语义。
- 不为了“代码看起来现代”而硬塞与当前项目不一致的新抽象、工具类或设计模式。
- 如果相邻模块已经存在统一函数命名、转换入口或提示入口，新增代码优先贴近相邻模块，不另起一套平行写法。

### 9. 父子窗体 / 弹窗协作规则

涉及父窗体打开子窗体、编辑弹窗、明细弹窗或业务处理弹窗时，默认保持职责边界清晰：

- 弹窗负责自身上下文接收、初始化、回显、校验、保存和关闭结果。
- 父窗体负责 `ShowDialog()` 后根据返回结果刷新列表、当前行、状态文本、按钮可用性或明细区。
- 不要把父窗体刷新职责塞进弹窗内部，也不要把弹窗保存细节散落到父窗体事件中。
- 同一次保存成功只允许一个位置弹成功提示；如果弹窗内部已经提示成功，父窗体返回后只刷新状态，不再重复弹提示。
- 父子窗体之间优先使用项目既有委托、返回值、上下文对象或 `DialogResult` 模式，不随意新增不一致的通信方式。

### 10. 事务边界与联动修改范围

当一个保存或状态变更动作跨多个 Model、BLL、数据表或业务对象时，优先尊重项目现有事务边界和调用链：

- 相邻模块已有事务或批量保存模式时，优先沿用，不轻易拆散为多个互不相关的保存调用。
- 字段贯穿 UI、Model、BLL、IDAL/DAL、SQL 脚本或配置时，按需求完整核对链路，不只改界面。
- 如果只是补显示列、查询条件或局部校验，不顺手重写整页保存流程、业务流程或分层结构。
- 涉及状态流转、主从表、批量操作、回传刷新时，优先拆成状态计算、数据保存、列表刷新、界面状态更新等职责函数。

### 11. 数据访问与 SQL 规则

三层访问默认遵循：

- UI 层只调用 BLL，不直接越过到 IDAL / SQLServerDAL。
- BLL 通过项目既有工厂或接口进入数据访问实现。
- UI 层可以组织 `strWhere` 等动态筛选条件，但只限于过滤片段；完整 SQL 主体写在数据访问层。
- 查询类方法按项目风格直接组织 `WHERE` 条件片段，不使用 `SqlParameter`，不额外维护 `parameters`；写操作仍默认参数化。
- 保存、更新、删除、状态变更、事务脚本等写操作默认使用参数化查询。
- 内部计数、存在性或分页条件方法若已有稳定的条件片段拼接方式，优先沿用项目风格，但不要把完整 SQL 主体上移到 UI 层。
- SQL 主体默认使用完整表名；是否使用表别名、字段别名，应贴近当前项目规则；无重名、无表达式语义需求时，不主动新增字段别名。
- 多表关联优先使用清晰的 `INNER JOIN` / `LEFT JOIN`，避免扩散旧式逗号联表写法，除非项目既有方法明确保持旧风格。
- 读取单值时优先使用 `GetSingle()`、`Exists()` 或项目等价方法，不为单字段返回 `DataSet`。
- 多行多列查询如果调用处只消费首个结果表，进入业务处理前先提取为 `DataTable` 局部变量，再统一使用该表；不要在后续循环、判空或绑定中反复保留 `DataSet` 并穿透 `Tables[0]`。只有确实需要多个结果表时，局部变量才保留 `DataSet`。
- `NULL` / `DBNull` 处理优先复用项目公共转换与空值处理能力，如 `Common.ConvertObject`、`Common.Tools.IsValueNull<T>(...)` 或相邻模块等价入口。
- 所有 `IDisposable` 对象必须使用 `using` 或等价可靠释放方式管理生命周期。

### 12. 代码风格与转换规则

默认遵循：

- 4 个空格缩进，不使用 Tab。
- Allman 花括号风格。
- `using` 顺序为 `System` → 第三方 → 项目内。
- 单行长度尽量控制在 120 字符内；明显影响可读性时主动换行。
- 函数 / 方法调用与定义参数少于 6 个时默认保持单行；达到或超过 6 个，或单行影响可读性时，再按参数换行。
- 字符串变量嵌入默认优先使用 `$""` 插值字符串；如果当前历史代码块大量使用 `string.Format(...)` 且保持一致更安全，则沿用原风格。
- 拼接 SQL 时禁止使用 `AppendFormat(...)`；统一使用 `Append($"...")` 或普通 `Append(...)`。
- 仅用于 SQL 拼接的枚举整型值不要先赋给局部变量；直接在 `$"..."` 中写 `(int)枚举值`。
- 三元运算符默认保持单行；如果单行装不下或可读性变差，改用 `if/else` 或临时变量，不把 `?`、`:` 拆到多行。
- 局部变量是否允许 `var` 以当前项目规则为准；如果项目明确禁止，则使用显式类型声明，并覆盖 `foreach`、`out var`、模式匹配等容易误用隐式类型的场景。
- 从 DataRow / DataTable / object 转换具体类型时，优先复用项目公共转换器，如 `Common.ConvertObject` 或相邻模块等价入口；不要到处散落 `Convert.ToInt32(...)`、`Convert.ToDecimal(...)` 等重复转换。
- 反向写入可空字段或处理 `NULL` / `DBNull` 时，优先复用项目公共空值处理入口，如 `Common.Tools.IsValueNull<T>(...)` 或相邻模块等价方法。
- 动软或项目模板生成的 `DataRowToModel(DataRow row)` 方法，若已有固定逐字段映射风格，新增字段应贴近原风格，不整体重写。
- 优先使用 guard clause 减少嵌套；常规业务方法嵌套过深时拆分职责函数或早返回。
- 捕获具体异常并提供明确错误信息；禁止空 `catch`、静默吞异常、无语义异常。

### 13. 注释与 `#region`

当任务涉及 C# 文件新增或修改时，只在本次改动范围内补充与职责匹配的注释：

- Model / DTO / 界面承载对象：新增类、枚举、属性、转换函数时，补充简短职责说明。
- BLL / IDAL / SQLServerDAL 新增公开方法：补充 `summary` 与必要参数说明。
- WinForms 窗体代码：对新增职责函数、关键流程入口、查询入口、打开子窗体、状态切换等函数补充适量注释。
- 不为了“统一风格”大面积补历史代码注释。

`#region` 只用于职责分组：

- 文件同时包含较多字段、职责函数、事件入口时，可按项目风格使用 `#region 字段和属性`、`#region 自定义函数`、`#region 事件处理` 或相邻模块既有命名。
- 构造函数与 `Load` 入口默认独立存在，不放入 region。
- 区块命名使用职责词或业务词，不使用“其他”“杂项”“临时处理”。
- 控件事件区只负责事件入口和界面动作分发，不把完整业务处理、保存逻辑或查询主流程直接堆进去。
- 文件较短、职责单一，或相邻窗体本来不用 region 时，不为了形式强行新增。
- 修改既有文件时，优先沿用原文件已有 region 层级、顺序和命名方式，不仅为整理结构而大面积重排历史代码。

### 14. 固定选项组规则

少量固定选项组默认使用普通 `Panel` 承载 `RadioButton` / `CheckBox`，不要默认使用 `FlowLayoutPanel`。

适用场景：

- 2~4 个固定数量单选项。
- 2~4 个固定数量复选项。
- 录入弹窗、维护页中只需稳定横向或纵向排布，且数量不会运行时动态变化的布尔选项组。

仅在选项数量运行时动态变化、明确需要自动换行或自适应流式排列时，才考虑 `FlowLayoutPanel`。
